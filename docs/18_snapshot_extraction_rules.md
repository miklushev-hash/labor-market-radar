# Правила извлечения структурированного слоя для radar_snapshot.json

## Назначение

Этот документ фиксирует, какие структурированные данные можно извлекать из готовых выпусков Labor Market Radar и как переносить их в `radar_snapshot.json`.

`radar_snapshot.json` не является новым хранилищем первичных данных. Он хранит только утвержденную агрегированную картину для дашборда и прототипа.

## Источники для извлечения

Допустимые основания:

1. готовый Markdown-файл регулярного выпуска;
2. финальные HTML/PDF из `presentations`;
3. листы `Недели`, `Месяцы`, `Кварталы`, `Полугодия`, `Годы` и `Журнал_событий` в `Хранилище.xlsx`;
4. сводный радар текущей ситуации, если он уже собран и явно утвержден пользователем.

Архивные версии не используются как актуальная основа, если пользователь отдельно не просит работать с архивом.

Сырые новости, непроверенные ссылки и незаполненные строки источников нельзя напрямую переносить в snapshot.

## Утвержденный набор структурированных данных

Минимальный snapshot должен содержать:

- `schema_version`;
- `snapshot_id`;
- `as_of_date`;
- `status`;
- `market_mode`;
- `short_answer`;
- `basis`;
- `indexes`;
- `signals`;
- `actions`;
- `limitations`;
- `updated_from`.

## Что извлекать из регулярного выпуска

### Из годового выпуска

- базовый режим года;
- устойчивые структурные тренды;
- макромеханизмы года;
- сильные подтвержденные сигналы;
- ограничения, с которыми рынок вошел в следующий год.

Годовой выпуск задает фон. Он не должен сам по себе резко двигать текущий snapshot, если после него уже есть более свежие квартальные, месячные или недельные основания.

### Из квартального выпуска

- изменение траектории года;
- сигналы, ставшие устойчивыми за квартал;
- что оказалось шумом;
- сравнения с прошлым кварталом, началом года и тем же кварталом прошлого года, если история есть.

Квартальный выпуск может менять режим рынка и индексы, если подтверждает сдвиг, а не одиночное событие.

### Из месячного выпуска

- последнее подтвержденное движение;
- новые усиливающиеся темы;
- изменение поведения работодателей, кандидатов и рынка труда;
- стратегические события месяца;
- ограничения по источникам.

Месячный выпуск чаще всего является главным источником актуального движения snapshot.

### Из недельного среза

- свежие сильные сигналы;
- значимые события ближайшего горизонта;
- ранние изменения риторики;
- явные ручные проверки.

Недельный срез не обязан менять snapshot. Если недельный выпуск слабый, короткий или праздничный, он может обновить только `last_reviewed_at`, `basis` или ограничения.

### Из сводного радара

Сводный радар может быть источником snapshot только если:

- он явно собран по запросу пользователя;
- перечисляет выпуски-основания;
- содержит объясненные индексы;
- отделяет сильные выводы от слабых сигналов;
- называет ограничения и ручные проверки.

## Правила извлечения сигналов

Каждый сигнал в snapshot должен иметь:

- `id`;
- `name`;
- `status`;
- `strength`;
- `evidence`;
- `meaning_now`;
- `basis_refs`.

Допустимые статусы:

- `confirmed` - подтвержден готовым выпуском и источниками;
- `supported` - поддержан несколькими основаниями, но не является жестким выводом;
- `watch` - находится на наблюдении;
- `manual_check_required` - нельзя обоснованно восстановить статус без ручной проверки.

Если в выпуске сигнал сформулирован осторожно, в snapshot нельзя повышать его до `confirmed`.

## Правила извлечения действий

Действие можно переносить в snapshot, только если оно следует из готового вывода выпуска или сводного радара.

Каждое действие должно иметь:

- `id`;
- `title`;
- `urgency`;
- `owner`;
- `basis_refs`.

Запрещено добавлять действия, основанные на личной догадке без связи с выпуском.

## Правила обработки отсутствующих данных

Если значение нельзя восстановить из готовых материалов:

- не выдумывать;
- ставить `manual_check_required`;
- добавить объяснение в `limitations`;
- не использовать это значение как основание для индекса.

## Проверка перед обновлением snapshot

Перед сохранением `radar_snapshot.json` нужно проверить:

1. JSON валиден;
2. все файлы в `basis` существуют;
3. каждый индекс имеет `score`, `label`, `explanation`, `basis_refs`;
4. каждый сильный сигнал имеет основание;
5. ограничения заполнены;
6. `updated_from` указывает на реальный документ;
7. дашборд не показывает устаревший fallback, если snapshot доступен.

## Что считается ошибкой

Ошибкой считается:

- индекс без объяснения;
- сигнал без основания;
- действие без связи с выпуском;
- ссылка на архив как на актуальный выпуск без решения пользователя;
- перенос первичного реестра источников в snapshot;
- запись сводного радара в `Хранилище.xlsx`;
- обновление snapshot по непроверенной новости.
