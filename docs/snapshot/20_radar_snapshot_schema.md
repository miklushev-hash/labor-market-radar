# Схема radar_snapshot.json

## Назначение

Этот документ фиксирует структуру `radar_snapshot.json` - файла, который превращает готовые аналитические выпуски Labor Market Radar в данные для дашборда.

Цепочка продукта:

```text
Обзоры объясняют
-> правила извлечения говорят, что брать
-> схема snapshot говорит, куда класть
-> radar_snapshot.json фиксирует данные
-> дашборд показывает
```

`radar_snapshot.json` не заменяет `Хранилище.xlsx` и не является реестром первичных источников. Он хранит только утвержденную агрегированную картину текущей ситуации.

## Связанные документы

- `18_snapshot_extraction_rules.md` - что извлекать из выпусков.
- `19_index_methodology.md` - как понимать и обосновывать числовые индексы.
- `20_radar_snapshot_schema.md` - как разложить извлеченные элементы по полям JSON.

## Расположение файла

Текущий рабочий snapshot хранится здесь:

```text
data/current_radar/radar_snapshot.json
```

Архивные версии при необходимости хранятся так:

```text
data/current_radar/archive/radar_snapshot_YYYY-MM-DD.json
```

Старые версии нельзя удалять без решения пользователя. Если файл заменяется вручную и архив не используется, прежнюю версию нужно переносить в `Корзина`.

## Обязательные верхнеуровневые поля

`radar_snapshot.json` должен содержать:

- `schema_version` - версия схемы, сейчас `1.0`;
- `snapshot_id` - уникальный идентификатор snapshot;
- `as_of_date` - дата, на которую актуальна картина;
- `created_at` - дата создания файла;
- `status` - статус готовности;
- `methodology_status` - статус методологии индексов;
- `dashboard_binding` - подключен ли дашборд к этому файлу;
- `market_mode` - краткое название текущего режима рынка;
- `short_answer` - управленческий ответ "и что теперь?";
- `basis` - выпуски и документы-основания;
- `indexes` - числовые индексы;
- `signals` - главные сигналы;
- `actions` - рекомендуемые действия;
- `limitations` - ограничения и ручные проверки;
- `updated_from` - документ или выпуск, из которого собран snapshot.

## Допустимые значения

### `status`

- `draft` - черновик, нельзя использовать как утвержденную картину;
- `pilot` - пилотный snapshot, допустим для прототипа и обсуждения;
- `approved` - утвержденная версия для дашборда;
- `archived` - историческая версия.

### `methodology_status`

- `expert_pilot` - экспертная методология, не математическая модель;
- `partially_validated` - часть индексов подтверждена историей и HR-последствиями;
- `mathematically_validated` - индексы имеют стабильные входные показатели и проверенную связь с HR-последствиями.

### `dashboard_binding`

- `not_connected` - дашборд пока не читает этот JSON;
- `connected_with_fallback` - дашборд читает JSON, но имеет fallback-данные;
- `connected_required` - дашборд работает только от JSON.

### Статусы сигналов

- `confirmed` - подтвержден готовыми выпусками и источниками;
- `supported` - поддержан несколькими основаниями, но не является жестким выводом;
- `watch` - сигнал на наблюдении;
- `manual_check_required` - статус нельзя восстановить без ручной проверки.

### Срочность действий

- `high` - нужно учитывать в ближайшем управленческом цикле;
- `medium` - нужно проверить и встроить в планирование;
- `low` - наблюдать, без немедленного действия.

## Схема индекса

Каждый объект в `indexes` должен содержать:

- `id` - технический идентификатор;
- `name` - человекочитаемое название;
- `score` - число от `0` до `100`;
- `label` - короткая интерпретация;
- `method` - способ получения значения;
- `explanation` - почему выбран этот диапазон и балл;
- `basis_refs` - ссылки на основания из `basis`;
- `confidence` - `low`, `medium` или `high`;
- `manual_check_required` - `true` или `false`.

Пока методология экспертная, `method` должен быть `expert_assessment`, а не `formula`.

## Схема сигнала

Каждый объект в `signals` должен содержать:

- `id`;
- `name`;
- `status`;
- `strength` - `weak`, `medium`, `strong`;
- `meaning_now`;
- `evidence`;
- `basis_refs`;
- `manual_check_required`.

## Схема действия

Каждый объект в `actions` должен содержать:

- `id`;
- `title`;
- `urgency`;
- `owner`;
- `rationale`;
- `basis_refs`.

## Приоритет выпусков

Если несколько выпусков дают разную картину, приоритет определяется так:

1. утвержденный сводный радар текущей ситуации, если он явно собран по запросу и содержит основания;
2. месячный выпуск как главный источник актуального движения;
3. квартальный выпуск как проверка устойчивости траектории;
4. годовой выпуск как фон и структурная рамка;
5. недельный срез как ранний сигнал, который сам по себе не должен резко менять индексы.

Недельный выпуск может обновить `basis`, `last_reviewed_at`, отдельный сигнал или ограничение. Он не должен резко менять режим рынка без подтверждения месячным или квартальным уровнем.

## Когда индекс можно менять

Индекс можно менять, если:

- появился новый готовый выпуск;
- новый выпуск подтверждает изменение, а не одиночный шум;
- изменение связано с тем, что измеряет индекс;
- есть объяснение, почему значение стало выше или ниже;
- ограничения и ручные проверки явно указаны.

Индекс нельзя менять:

- по сырой новости;
- по одному слабому недельному сигналу;
- если основание противоречиво и не разобрано;
- если нельзя указать `basis_refs`;
- если изменение основано только на ощущении аналитика.

## Шаблон заполнения

Минимальный шаблон:

```json
{
  "schema_version": "1.0",
  "snapshot_id": "radar_YYYY-MM-DD",
  "as_of_date": "YYYY-MM-DD",
  "created_at": "YYYY-MM-DD",
  "status": "pilot",
  "methodology_status": "expert_pilot",
  "dashboard_binding": "not_connected",
  "market_mode": {
    "id": "short_mode_id",
    "label": "Краткое название режима"
  },
  "short_answer": "Краткий управленческий вывод.",
  "basis": [],
  "indexes": [],
  "signals": [],
  "actions": [],
  "limitations": [],
  "updated_from": []
}
```

## Проверка корректности

Перед использованием snapshot нужно проверить:

1. JSON валиден;
2. все файлы в `basis` существуют;
3. каждый индекс имеет `score`, `label`, `explanation`, `basis_refs`;
4. каждый сильный сигнал имеет основание;
5. ограничения заполнены;
6. `updated_from` указывает на реальные документы;
7. если `dashboard_binding` не равен `not_connected`, дашборд действительно читает JSON, а не зашитый массив.

## Правило для дашборда

Промышленная версия дашборда должна загружать данные из `radar_snapshot.json`.

Если дашборд использует зашитый демонстрационный массив, он считается прототипом. В этом случае в snapshot должно стоять:

```json
"dashboard_binding": "not_connected"
```

Переход дашборда на чтение JSON является отдельной технической задачей после согласования схемы.
