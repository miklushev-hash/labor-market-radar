# Техническая архитектура дашборда Labor Market Radar

## Назначение документа

Этот документ фиксирует, как данные из регулярных выпусков должны попадать в дашборд и локальный прототип Labor Market Radar.

Главное архитектурное решение:

```text
Годовой / квартальный / месячный / недельный выпуск
        ↓
структурированный слой данных
        ↓
radar_snapshot.json
        ↓
дашборд / прототип

Сводный радар текущей ситуации
        ↑
        опционально собирается из того же структурированного слоя
```

Сводный радар текущей ситуации не является обязательным промежуточным звеном между выпусками и дашбордом. Он может быть, а может не быть.

Дашборд должен жить от `radar_snapshot.json`, а не от текста сводного радара.

## Цель дашборда

Дашборд должен показывать текущую управленческую картину рынка труда:

- общий режим рынка;
- индексы по шкале `0-100`;
- все 5 индексов на главном экране;
- всплывающие подсказки к индексам простым языком;
- пояснение внутри каждой карточки индекса в секции "Это означает, что";
- блок "Что произошло за период" с фундаментальными фактами периода;
- блок "Яркие события ИТ-рынка" с отобранными ИТ-событиями;
- блок "Зарплатный фон" с динамикой, а не одиночной общей цифрой;
- главные сигналы;
- доказательность и ограничения;
- что это значит для работодателя;
- действия;
- связь с выпусками-основаниями;
- в перспективе — внутренний контур практик, ролей, компетенций и кадровых рисков.

Дашборд не должен сам принимать аналитическое решение из сырых текстов. Он должен показывать последнюю утвержденную структурированную версию.

На пользовательском экране техническая терминология должна быть сведена к минимуму. Слово `snapshot`, путь к JSON и служебные статусы допустимы в документации и диагностике, но основной экран должен говорить человеческим языком: "дата оценки", "источник данных", "предварительная экспертная оценка", "данные загружены из рабочего файла".

## Что остается в выпусках

Регулярные выпуски остаются human-readable артефактами:

- Markdown;
- HTML;
- PDF;
- приложение по источникам;
- связка с `Хранилище.xlsx`.

Они отвечают на вопрос:

> Что произошло за период и на каких источниках это основано?

Форматы выпусков остаются прежними:

- неделя;
- месяц;
- квартал;
- полугодие;
- год.

## Что появляется дополнительно

После готового выпуска или после ручной управленческой интерпретации может формироваться машинный слой:

- `radar_snapshot.json`

Это не новое хранилище первичных данных. Это компактное представление текущей аналитической картины для интерфейса.

## Роль `radar_snapshot.json`

`radar_snapshot.json` — это единственный источник данных для дашборда в локальном прототипе.

Он должен хранить:

- дату актуальности;
- список выпусков-оснований;
- общий режим рынка;
- числовые индексы;
- подсказки к индексам;
- ключевые выводы периода;
- яркие события ИТ-рынка;
- зарплатную динамику;
- главные сигналы;
- действия;
- ограничения;
- статус утверждения;
- источник последнего обновления.

Прототип не должен держать эти данные зашитыми внутри HTML. HTML отвечает за интерфейс, JSON — за данные.

## Где хранить snapshot

На текущем локальном этапе рекомендуемое место:

- `data/current_radar/radar_snapshot.json`

Для удобства прототипа допускается копия:

- `prototypes/labor-market-radar/data/radar_snapshot.json`

Но рабочим источником истины лучше считать файл в `data/current_radar`.

Если прототипу нужна локальная копия, она должна обновляться из рабочего файла, а не жить как отдельная версия правды.

## Роль сводного радара текущей ситуации

Сводный радар текущей ситуации — опциональный human-readable артефакт.

Он нужен, когда пользователь просит управленческий ответ:

> И что теперь?

Он может использовать:

- готовые выпуски;
- `radar_snapshot.json`;
- экспертную интерпретацию;
- дополнительные явные ограничения.

Но он не является обязательным этапом для обновления дашборда.

Правильная зависимость:

```text
radar_snapshot.json
  ├─ обновляет дашборд
  └─ может использоваться для сводного радара по запросу
```

Если сводный радар меняет управленческую оценку, после него можно обновить `radar_snapshot.json`. Но это отдельное действие, а не автоматическое правило.

## Потоки данных

### Поток 1. Регулярный выпуск обновляет дашборд

```text
готовый выпуск
  → извлечение структурированных выводов
  → обновление radar_snapshot.json
  → дашборд показывает новую версию
```

Этот поток подходит, когда новый выпуск дает достаточно оснований для изменения текущей картины.

### Поток 2. Регулярный выпуск не меняет дашборд

```text
готовый выпуск
  → проверка структурированных выводов
  → решение: текущая картина не изменилась
  → radar_snapshot.json не меняется или обновляется только metadata
```

Это важно для недельных срезов: не каждое короткое окно должно двигать индексы.

### Поток 3. Сводный радар по запросу

```text
готовые выпуски + radar_snapshot.json
  → сводный радар текущей ситуации
  → при необходимости обновление radar_snapshot.json
  → дашборд показывает утвержденную версию
```

Сводный радар здесь не обязателен. Он появляется только по запросу.

## Правила обновления snapshot

### 1. Snapshot обновляется только после готового основания

Основанием может быть:

- готовый регулярный выпуск;
- сводный радар по запросу;
- ручное экспертное решение, явно оформленное в документе.

Нельзя обновлять `radar_snapshot.json` по сырой новости, которая еще не прошла проверку.

### 2. Snapshot не обязан обновляться после каждого выпуска

Если выпуск не меняет текущую картину, можно:

- не менять snapshot;
- либо обновить только `last_reviewed_at`, `basis` или комментарий.

Это особенно важно для недельных срезов.

### 3. Каждое число должно иметь объяснение

Для каждого индекса обязательны:

- `score`;
- `label`;
- `explanation`;
- `tooltip`;
- `basis_refs`.

Число без объяснения считается технически некорректным.
Поле `explanation` должно выводиться на главном экране прямо в карточке индекса в нижней секции "Это означает, что". Отдельный блок с горизонтальными шкалами для индексов не используется, потому что индексы отражают разные аспекты ситуации и не должны восприниматься как напрямую сопоставимые между собой показатели.

### 4. Snapshot не заменяет `Хранилище.xlsx`

`Хранилище.xlsx` остается хранилищем данных, которые собираются при формировании обзора.

`radar_snapshot.json` хранит только текущую агрегированную картину для интерфейса.

В snapshot нельзя переносить первичный реестр источников, статусы всех источников и операционные строки периода.

## Версионирование

Минимально:

- текущий файл: `data/current_radar/radar_snapshot.json`;
- архивные версии: `data/current_radar/archive/radar_snapshot_YYYY-MM-DD.json`.

При обновлении snapshot старую версию лучше переносить в архив или `Корзина` по правилам проекта, но не удалять.

## Как прототип должен читать данные

На текущем локальном этапе:

1. `index.html` загружает `../../data/current_radar/radar_snapshot.json`.
2. Если загрузка не удалась, прототип показывает явное fallback-предупреждение, что актуальный snapshot недоступен.
3. В интерфейсе явно отображается:
   - дата snapshot;
   - статус;
   - основание;
   - индексы;
   - ограничения.

Желательное поведение:

- интерфейс не ломается без JSON;
- пользователь видит, что данные демонстрационные, если JSON не найден;
- при найденном JSON дашборд показывает реальные значения из snapshot.

## Что меняется в прототипе

Текущий прототип в `prototypes/labor-market-radar/index.html` уже читает `data/current_radar/radar_snapshot.json` через относительный путь `../../data/current_radar/radar_snapshot.json`.

Текущий статус:

1. `data/current_radar/radar_snapshot.json` создан;
2. зашитый массив данных в HTML удален;
3. дашборд показывает дату snapshot, статус, методологию, индексы, сигналы, действия, ограничения и выпуски-основания;
4. fallback сохранен только как явное предупреждение на случай ошибки загрузки JSON;
5. `dashboard_binding` в snapshot должен быть `connected_with_fallback`.

## Проверки перед тем, как считать дашборд актуальным

Перед использованием дашборда нужно проверить:

1. `radar_snapshot.json` валидный JSON;
2. `schema_version` поддерживается прототипом;
3. `as_of_date` соответствует последней утвержденной оценке;
4. все индексы имеют `score`, `label`, `explanation`, `tooltip`;
5. все сильные сигналы имеют основания;
6. ограничения заполнены;
7. дашборд показывает дату и статус snapshot;
8. дашборд не берет данные из устаревшего fallback, если рабочий JSON доступен.
9. на главном экране показаны все 5 индексов;
10. у каждой карточки индекса есть нижняя секция "Это означает, что" с человеческим объяснением;
11. пользовательский экран не показывает технические слова и пути как основное содержание.

## Отношение к промышленной архитектуре

В промышленном запуске `radar_snapshot.json` станет не ручным файлом, а ответом API или материализованным представлением из базы.

Текущая локальная схема нужна как мост:

```text
файлы проекта → структурированный JSON → дашборд
```

Промышленная схема будет такой:

```text
источники / проверки / выпуски / внутренний контур
        ↓
аналитические сущности в БД
        ↓
Dashboard API
        ↓
веб-дашборд
```

Но логика остается той же: дашборд показывает утвержденный структурированный слой, а не сам придумывает выводы из сырых материалов.
