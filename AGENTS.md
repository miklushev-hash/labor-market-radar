# AGENTS.md

Этот файл является постоянно развивающейся базой знаний проекта для Codex и других агентных запусков. Его нужно обновлять после значимых изменений продукта, архитектуры, документации, правил работы или пользовательских предпочтений.

## 1. Краткое описание проекта

Проект `Обзор РТ` / `Labor Market Radar` - аналитический продукт для регулярного наблюдения за рынком труда, с главным фокусом на рынке труда ИТ в России и практических выводах для работодателя.

Продукт объединяет:

- регулярные обзоры: недельный срез, месячный, квартальный, полугодовой и годовой выпуск;
- рабочие реестры источников, компаний и стратегических событий;
- `Хранилище.xlsx` как операционную книгу фиксации проверок и событий;
- сводный радар текущей ситуации по запросу;
- `radar_snapshot.json` как структурированный слой данных;
- публичный GitHub Pages-дашборд, который читает snapshot.

Текущий статус продукта: пилотная версия. Внешний контур уже работает через выпуски, snapshot и публичный дашборд. Внутренний HR-контур компании пока не подключен и требует отдельного согласования с HRD.

## 2. Технологии и архитектурные принципы

### Используемые технологии

- Markdown - основная документация, шаблоны и текстовые выпуски.
- XLSX - рабочие реестры и операционное хранилище проверок.
- JSON - структурированный snapshot для дашборда.
- HTML/CSS/JavaScript - публичная витрина и прототип дашборда.
- PDF/HTML - финальные версии выпусков в `presentations`.
- Git/GitHub - версионирование проекта.
- GitHub Pages - публичная публикация корневого `index.html` и прототипа дашборда.

### Архитектурные принципы

- Текущий рабочий контур остается файловым, пока обновление делает один человек и частота обновления не выше примерно одного раза в неделю.
- `Хранилище.xlsx` хранит операционные данные проверок и событий, но не усложняется логикой радара.
- `radar_snapshot.json` является витринным слоем данных для дашборда, а не новым первичным хранилищем.
- Дашборд не должен хранить актуальные данные внутри HTML; он должен читать `data/current_radar/radar_snapshot.json`.
- Публичный сайт GitHub Pages публикуется из корня репозитория, чтобы не держать дубликаты `presentations` внутри `docs`.
- Сводный радар текущей ситуации является опциональным управленческим артефактом по запросу, а не обязательным слоем после каждого выпуска.
- Внутренний HR-контур должен подключаться отдельным этапом; до этого внутренние данные нельзя имитировать как реальные.

## 3. Структура проекта

- `config/` - конфигурационные материалы проекта.
- `data/` - данные и подготовленные материалы по периодам; текущий snapshot лежит в `data/current_radar/radar_snapshot.json`.
- `docs/` - проектная документация, регламенты, архитектура, правила выпусков.
- `docs/snapshot/` - документы, связанные со snapshot: правила извлечения, методика индексов, схема JSON.
- `docs/application/` - материалы, связанные с приложением/локальными интерфейсами.
- `local_tmp/` - локальные временные результаты, презентации и промежуточные файлы; Git игнорирует эту папку.
- `presentations/` - актуальные HTML/PDF-версии выпусков и управленческих материалов. Это основной источник актуальных публичных материалов.
- `prototypes/labor-market-radar/` - публичный прототип дашборда Labor Market Radar.
- `source/` - личные рабочие материалы пользователя, заметки, черновики и вспомогательные файлы. Папка игнорируется Git и не считается частью публикуемого продукта.
- `taxonomy/` - словари и классификации.
- `templates/` - шаблоны выпусков и документов.
- `Корзина/` - локальная папка для лишнего, спорного, замененного или устаревшего. Git игнорирует ее.
- `index.html` - корневая публичная страница GitHub Pages.
- `README.md` - входная навигация по проекту и правилам.
- `Главный_документ_продукта.md` - главная продуктовая логика, форматы, контуры, критерии готовности.
- `Хранилище.xlsx` - рабочее хранилище проверок по периодам и `Журнал_событий`.
- `Источники данных.xlsx` - реестр источников, который нужно проходить при сборке выпусков.
- `Каркас системы выпусков.xlsx` - управляющая матрица форматов.
- `Компании стратегического наблюдения.xlsx` - реестр компаний стратегического наблюдения.
- `События стратегического наблюдения.xlsx` - справочник/материалы по событиям; операционный журнал событий перенесен в `Хранилище.xlsx`.

## 4. Правила разработки и кодирования

- Работать только внутри текущего проекта, если пользователь явно не попросил другое.
- Не трогать чужие локальные изменения без явного запроса.
- Не коммитить пользовательские рабочие файлы из `source`; папка `source/` должна оставаться локальной и игнорироваться Git.
- Не удалять файлы безвозвратно: лишнее и заменяемое переносить в `Корзина`.
- Из `Корзина` ничего не удалять ни при каких условиях. Чистить `Корзина` имеет право только пользователь.
- `Корзина/`, `local_tmp/`, временные файлы и lock-файлы не должны попадать в Git.
- Не использовать архивные версии выпусков как актуальные, если пользователь отдельно не попросил работать с архивом.
- Актуальные версии обзоров брать из `presentations/`, а не из архивных копий.
- Для GitHub Pages не создавать дубли `docs/presentations`; публикация идет из корня репозитория.
- Для дашборда актуальные данные должны идти из `data/current_radar/radar_snapshot.json`, а не из зашитого массива в HTML.
- Готовые пользовательские HTML/PDF-выпуски в `presentations` должны называться по-русски: `Годовой обзор <год>`, `Квартальный обзор <N квартал> <год>`, `Месячный обзор <месяц> <год>`, `Недельный срез <ДД.ММ.ГГГГ-ДД.ММ.ГГГГ>`, `Сводный радар <ДД.ММ.ГГГГ>`.
- При ручных правках файлов использовать аккуратные, точечные изменения; не делать косметические массовые переписывания без причины.
- Для поиска использовать `rg`, если доступен.

## 5. Правила проектирования новых функций

- Новая функция должна быть связана с продуктовой логикой: регулярные выпуски, snapshot, дашборд, внутренний контур, источники или управленческие выводы.
- Перед изменением логики проверить README, главный документ продукта, матрицу выпусков и профильные документы в `docs`.
- Если функция меняет выпуск, формат, snapshot, дашборд или хранилище, задача не считается завершенной без обновления связанной документации.
- Не усложнять `Хранилище.xlsx`: оно хранит данные, собираемые при формировании обзоров, а не всю аналитику продукта.
- Для текущего этапа JSON допустим как переходный слой данных. Переход к БД нужен при появлении внутреннего HR-контура, истории, связей, ролей доступа, нескольких пользователей или регулярных автоматических обновлений.
- Внутренний контур компании не подменять демонстрационными данными без явной маркировки.
- Для стратегических событий использовать фильтр значимости: событие должно иметь объяснимую цепочку влияния на рынок труда, ИТ-рынок, работодателей, стратегических клиентов или отраслевые правила игры.
- Широкий мониторинг допустим только как широкий вход; выход продукта должен оставаться узким и управленчески полезным.

## 6. Требования к тестированию и проверкам

### Для документации

- Проверять отсутствие битых ссылок после переноса файлов.
- Проверять, что старые названия, устаревшие пути и архивные ссылки не используются как актуальные.
- После изменений в архитектуре, правилах, форматах, путях, данных или процессах обязательно сверять README, главный документ продукта и профильные документы.
- Если при сверке найдено расхождение, устаревшая формулировка, старый путь или неполное описание новой логики, документацию нужно обновить в рамках той же задачи до финального отчета.

### Для XLSX

- Проверять листы, заголовки, заполненность, внутреннюю логику и соответствие формату.
- Перед изменениями ключевых XLSX, если пользователь просит, делать копию в `Корзина`.
- Если статус источника нельзя восстановить из обзора или приложения, явно ставить ручную проверку, не выдумывать результат.

### Для выпусков

- Проверять Markdown, HTML и PDF на согласованность.
- PDF проверять отдельно по PDF-слою, когда пользователь просит тотальную проверку.
- Готовым выпуском считается комплект: заполненное хранилище, Markdown, HTML, PDF, приложение/основания по источникам, отметка использованных источников.

### Для snapshot и дашборда

- `radar_snapshot.json` должен быть валидным JSON.
- Все пути в `basis` и `updated_from` должны существовать.
- Каждый индекс должен иметь `score`, `label`, `explanation`, `basis_refs`.
- Каждый сильный сигнал должен иметь основание.
- Ограничения должны быть заполнены.
- Дашборд должен читать snapshot через относительный путь и показывать fallback-предупреждение, если JSON недоступен.
- Перед публикацией проверять локально, что HTML содержит `fetch(SNAPSHOT_URL)` и не содержит старого `const data = {...}`.

Документация является частью продукта. После каждой значимой задачи нужно:

1. Проверить, какие документы затронуты.
2. Обновить связанные документы.
3. Обновить `AGENTS.md`, если появилось новое правило, предпочтение пользователя или проектное решение.
4. В финальном ответе кратко назвать, что изменено.

## User Preferences

- Пользователь предпочитает прямые, понятные объяснения без технического тумана.
- Пользователь не является техническим специалистом; перед техническими действиями нужно коротко объяснять человеческим языком, что будет сделано и зачем, а после действия - что было сделано и что это означает для проекта.
- Для сложных технических понятий нужны простые аналогии: JSON как заполненная карточка, snapshot как слепок данных, структура как правила перевода.
- Пользовательские экраны должны говорить человеческим языком: не выводить технические слова вроде `snapshot`, `JSON` или служебные статусы как основной текст, если их можно заменить понятной формулировкой.
- Ключевые выводы лучше начинать с утвердительной формулировки: сначала сказать, как выглядит ситуация, затем назвать ограничения.
- Рекомендации в продукте должны быть прикладными и проверяемыми, а не общими фразами вроде "нанимать точнее".
- Пользователь хочет понимать, что именно происходит и почему, а не просто получать выполненное действие.
- При сомнении лучше явно объяснить статус: пилот, экспертная оценка, ручная проверка, не математическая модель.
- Пользователь не хочет удалять лишнее безвозвратно; лишнее переносится в `Корзина`.
- `Корзина` принадлежит пользователю: агент не чистит ее никогда.
- Пользователь хочет, чтобы все продуктовые решения фиксировались в документации.
- Пользователь предпочитает не усложнять `Хранилище.xlsx`; там должны быть только данные, собираемые при формировании обзора.
- Пользователь хочет избегать дублей, особенно для GitHub Pages и `presentations`.
- Пользователь хочет видеть названия готовых выпусков на русском языке; англоязычные служебные имена допустимы только для внутренних Markdown/шаблонов, но не для финальных HTML/PDF-обзоров.
- Пользователь хочет, чтобы публичный продукт можно было показывать коллегам, но с честной маркировкой пилотного статуса.
- Пользователь принимает экспертную методологию индексов как временный пилотный этап, но хочет явно фиксировать, когда она должна перейти к математически подтвержденной модели.
- После нового выпуска пользователь может спросить: "есть ли основания для обновления snapshot/дашборда?" Ответ должен опираться на `docs/snapshot/*` и давать решение: обновлять полностью, частично или не обновлять.
- Пользователь часто просит тотальную проверку; такая команда означает чтение каждой строки доступных файлов, включая XLSX, DOCX, PDF, HTML, JSON, MD и проверку логики/ссылок/противоречий.
- Рабочие файлы пользователя в `source` нельзя трогать без явной просьбы; вся папка `source/` является личной зоной заметок пользователя и не должна попадать в Git.

## Project Learnings

- Текущий продукт реализован на файловых данных и JSON, потому что обновление делает один человек и частота обновления невысокая.
- `radar_snapshot.json` - текущий витринный слой данных для дашборда. Он не заменяет `Хранилище.xlsx`.
- Дашборд уже подключен к `data/current_radar/radar_snapshot.json` и имеет статус `connected_with_fallback`.
- Текущие индексы имеют статус `expert_pilot`: это объяснимая экспертная оценка, не математически валидированная модель.
- Переход к математически подтвержденной модели нужен после накопления истории snapshot/выпусков, определения стабильных входных показателей и проверки связи индексов с HR-последствиями.
- Переход к БД нужен не сейчас, а при подключении внутреннего HR-контура, истории, связей, ролей доступа, нескольких пользователей или автоматического сбора.
- Наиболее вероятный будущий тип БД для внутреннего контура - PostgreSQL; snapshot при этом может остаться витринным JSON/API-слоем.
- GitHub Pages публикуется из корня репозитория. Корневой `index.html` является витриной продукта.
- Публичный дашборд доступен по `https://miklushev-hash.github.io/labor-market-radar/prototypes/labor-market-radar/`.
- `docs/presentations` и `docs/prototypes` как дубли не нужны при публикации из корня.
- `Корзина/` исключена из Git и не должна попадать на GitHub.
- `voice_dictation_app` удален из проекта и GitHub как не относящийся к продукту.
- Пустая `.agents` была служебной папкой среды Codex, удалена локально; если среда создаст ее снова, это не часть продукта.
- `source/` переведена в локальную пользовательскую зону и добавлена в `.gitignore`; ранее отслеживаемые файлы из этой папки нужно снять с Git-учета без удаления с диска.
- Актуальные HTML/PDF-выпуски в `presentations` переименованы в русскоязычный формат; будущие готовые обзоры должны следовать этой схеме именования.

## Post-Task Checklist

После каждой значимой задачи:

1. Проверить `git status` и не включить в коммит пользовательские рабочие файлы.
